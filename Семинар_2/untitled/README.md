# Система Управления Спутниками (ООП & SOLID)

Этот проект представляет собой реализацию консольной системы управления спутниковой группировкой. Проект разработан для демонстрации применения принципов объектно-ориентированного программирования (ООП) и SOLID на языке Java.

## Описание проекта

Система моделирует работу космической группировки, состоящей из различных типов спутников:
*   **Спутники связи (CommunicationSatellite)** - отвечают за передачу данных с определенной пропускной способностью.
*   **Спутники дистанционного зондирования Земли (ImagingSatellite, ДЗЗ)** - занимаются фотосъемкой поверхности Земли с заданным разрешением.

Все спутники объединяются в **Спутниковую группировку (SatelliteConstellation)**, которая позволяет централизованно управлять ими (добавлять, активировать, запускать миссии).

## Принципы ООП

В проекте используются классические принципы ООП:
1.  **Абстракция:** Наличие базового абстрактного класса `Satellite` с общим контрактом `performMission()`.
2.  **Инкапсуляция:** Сокрытие внутреннего состояния объектов. Доступ к батарее и логике включения осуществляется через специальные методы.
3.  **Наследование:** `CommunicationSatellite` и `ImagingSatellite` наследуют базовую логику от `Satellite`.
4.  **Полиморфизм:** Класс `SatelliteConstellation` управляет списком всех спутников (типа `Satellite`) и вызывает уникальную для каждого класса реализацию `performMission()`.

## Принципы SOLID (Семинар 2)

В ходе рефакторинга код был адаптирован под принципы SOLID:

1.  **SRP (Принцип единственной ответственности):** Изначальный класс `Satellite` был перегружен. Теперь ответственность за энергию вынесена в отдельный класс `EnergySystem`, а за состояние активации — в `SatelliteState`. Сам спутник управляет только связями и миссией.
2.  **OCP (Принцип открытости/закрытости):** Благодаря выделению `EnergySystem` и `SatelliteState`, мы можем легко добавлять новые реализации спутников (например, метеоспутники), не трогая базовый класс и систему работы с батареей. Спутники открыты для расширения, но их базовое ядро закрыто для изменения.
3.  **LSP (Принцип подстановки Барбары Лисков):** Любой объект класса-наследника (`ImagingSatellite` или `CommunicationSatellite`) может быть использован везде (например, в списке группировки `SatelliteConstellation`), где ожидается базовый тип `Satellite`. Вызов методов базового класса у наследников не нарушает логику программы.

### Дальнейшее развитие (ISP и DIP)
*   **ISP (Принцип разделения интерфейсов):** Можно выделить интерфейсы вроде `DataTransmitter` (методы передачи) или `EarthObserver` (съемка), чтобы классы реализовывали только то, что им действительно нужно, и не тащили лишнего.
*   **DIP (Принцип инверсии зависимостей):** Сейчас в классе `Satellite` компоненты `EnergySystem` и `SatelliteState` создаются явно (жесткая зависимость). В будущем можно будет передавать имплементации интерфейсов, например, `IEnergySystem` извне при создании (через DI/конструктор), что позволит подменять солнечные батареи на ядерные реакторы без изменения кода спутника.

## Как запустить

1. Откройте проект в вашей IDE (рекомендуется IntelliJ IDEA).
2. Запустите класс `Main.java` (`src/main/java/org/example/Main.java`).
3. В консоль выведется симуляция создания группировки "RU Basic", распределения заряда и выполнения задач спутниками связи и ДЗЗ.
